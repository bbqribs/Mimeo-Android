## 1. Minimum target

T-AND-DIST-MIN-1 is done when all of the following are true:

1. **Signed release APK.** `.\gradlew.bat :app:assembleRelease` produces an APK
   signed with a dedicated household release key (not a debug key).
2. **Versioned APK artifact.** Every household-distributed build is a named,
   dated artifact the operator keeps (naming in §4); "whatever APK is lying
   around" is not a distribution channel.
3. **`versionCode` increments on every household-distributed build.** No
   exceptions, no reuse, no decrements — Android refuses in-place downgrades,
   so a skipped bump strands member devices (§5.4).
4. **`versionName` readable by operator and member.** A human string (for
   example `0.4.0`) visible in Android's app-info page and, after
   T-AND-DIST-MIN-1, in the app's Settings screen.
5. **Documented sideload install/update path.** The runbook in §5, executable
   by the operator with a member's phone in hand, no developer tooling on the
   member side.
6. **No Android Studio required on the member side.** Android Studio (or a
   Gradle checkout) exists only on the operator's build machine. A member
   device never needs a cable to a dev machine; `adb` is an operator
   convenience, never a member requirement.

## 2. Explicit non-goals

Work on these must not be justified by this ticket:

- **No Play Store / Play Console** (no Play App Signing, no store listing, no
  review process).
- **No public beta** — distribution is operator-to-household-member only.
- **No automatic update service** (no in-app updater, no update-check
  endpoint). Updates are operator-delivered per §5.2.
- **No crash-reporting SDK.** Problem reports remain the in-app opt-in flow.
- **No app-store privacy work** (data-safety forms, privacy policy URLs).
- **No CI/CD requirement.** Builds run on the operator's machine. If a trivial
  script helps (§8), fine; a pipeline is out of scope.
- No OAuth/MFA, no household GO claim, no backend changes — those belong to
  other lanes of the household queue.

## 3. Signing model

**One release keystore, operator-held, never in the repo.**

- **Where the key lives.** A dedicated keystore file
  (`mimeo-android-release.jks`) generated by the operator with `keytool`,
  stored **outside every git working tree** — recommended location: the same
  operator-controlled secrets area used for the backend key escrow (the T-KEY-1
  escrow procedure in the Mimeo repo), e.g. a `mimeo-secrets` folder that is
  already part of the off-host backup/escrow rhythm. The build reads its path
  and passwords from an untracked `keystore.properties` file in the Android
  repo root (template committed as `keystore.properties.example`; real file
  gitignored).
- **Who holds it.** The operator only. Members never see, hold, or need the
  key. There is exactly one signing identity for the household app.
- **What must never be committed:** the keystore file (`*.jks`, `*.keystore`),
  `keystore.properties` (paths + passwords), and passwords in any form —
  including in Gradle files, PR text, release notes, logs, or screenshots.
  T-AND-DIST-MIN-1 adds `.gitignore` entries for all of these **before** the
  key is generated, and the Gradle wiring must reference only the properties
  file, never inline credentials.
- **How to back it up.** Two independent copies minimum:
  1. the keystore file included in the existing off-host backup/escrow set
     (alongside the backend key escrow, same freshness rhythm);
  2. keystore + key passwords stored in the operator's password manager.
  Record the key's SHA-256 certificate fingerprint
  (`keytool -list -v -keystore ...`) in the release-notes file header so a
  future "is this the right key?" question is answerable without guessing.
- **If the key is lost** (and both backups are gone): no future build can
  update existing installs in place. Recovery is: generate a new key, bump
  `versionCode`/`versionName`, then on **every member device** uninstall the
  app and fresh-install the new APK. Server-side data (library, progress,
  playlists) survives — it lives on the backend and returns after sign-in.
  Device-local state (offline article cache, unsynced pending progress, local
  Up Next session, settings) is lost. Annoying but survivable at household
  scale; the backups exist so this never happens.
- **Interaction with reinstall/update.** Android's rule: an APK installs *over*
  an existing app only if (a) signed by the same key and (b) `versionCode` is
  equal or higher. Same key + higher code = clean in-place update, all app data
  preserved. Different key (e.g. a debug build or a lost-key rebuild) =
  `signature mismatch` error; the only path is uninstall-then-install, with the
  local-data loss above. This is why the key is backed up and why debug builds
  never go to member devices.

## 4. Versioning policy

- **`versionCode` discipline.** Monotonically increasing integer, bumped by +1
  in the release PR of every build intended for a member device. Never reused,
  never decreased, never bumped "speculatively" without a release. Current code
  is `1`; the first household release ships as `versionCode = 2`. Operator
  test-only builds that never touch a member device don't need a bump, but if
  in doubt, bump — codes are free, stranded devices are not.
- **`versionName`.** Human-readable `MAJOR.MINOR.PATCH`. Suggested first
  household release: `0.4.0` (continues the README's v0.2/v0.3 narrative;
  operator may choose differently — the invariant is that it changes whenever
  `versionCode` does and is meaningful when a member reads it aloud over the
  phone). Minor bump for feature releases, patch bump for fix-only releases.
- **Release artifact naming.**
  `mimeo-android-v<versionName>-vc<versionCode>-release.apk`
  (example: `mimeo-android-v0.4.0-vc2-release.apk`). The operator renames the
  Gradle output (`app/build/outputs/apk/release/app-release.apk`) to this form
  and keeps every distributed artifact in one operator-controlled folder
  (suggested: alongside the keystore in the secrets/artifacts area, included in
  off-host backup). Old artifacts are how a bad release gets diagnosed; keep
  at least the last three.
- **Minimal release note contents.** A `RELEASE_NOTES.md` at the Android repo
  root (created by T-AND-DIST-MIN-1; none exists today), newest entry first.
  Each entry:
  - versionName, versionCode, date;
  - artifact filename;
  - 2–6 plain-English bullets of what changed (member-visible first);
  - **backend-contract note**: either `Backend: no contract dependency` or
    `Backend: requires <endpoint/behavior> deployed (merged in Mimeo PR #N)` —
    a build must not be distributed while its backend dependency is undeployed
    on the household runtime;
  - known issues, if any member-visible ones exist.
- **"Clients affected" note.** When a *backend* change breaks or degrades older
  Android builds, that backend PR (per the CONTRACT CHANGE rule) should name
  the minimum Android versionCode that copes; the next Android release note
  echoes it ("members on vc2 must update before X works"). At household scale
  this is one sentence, not tooling.

## 5. Household install/update runbook

Written for the operator; the member-facing half stays in the household
GETTING_STARTED doc ("the operator installs or helps you install"). All flows
assume the signed release APK from §4, delivered to the phone by any trusted
household channel (USB cable + file copy, home NAS, or messaging app the
household already uses). Never publish the APK anywhere public.

### 5.1 First install on a trusted member device

1. Confirm the device is on the tailnet (Tailscale installed, signed in) — per
   the household getting-started doc, this comes first.
2. Get the APK onto the phone (cable copy to `Downloads`, or send via the
   household channel; the file arrives in the member's Files/Downloads).
3. Tap the APK in the Files app. Android will warn about unknown apps — see
   §5.3. Tap through Install.
4. If Play Protect interposes ("app from unknown developer"), choose
   **Install anyway** (`More details` → `Install anyway` on some builds). This
   is expected for any non-Play app and is safe here: the operator built and
   signed this APK.
5. Open Mimeo, set the server address if prompted
   (`https://beh-august2015.taildacac5.ts.net`), and have the member sign in
   with the username/password they set at `/welcome` (phone-browser step first
   if they haven't — per GETTING_STARTED Step 2/4).
6. Verify the version (§5.5) and log which versionCode this device now runs.

### 5.2 Updating over an older APK

1. Deliver the new APK to the phone (same channel as first install).
2. Tap it and Install — Android recognizes it as an update to the existing
   app. **Do not uninstall first**: an in-place update preserves sign-in,
   settings, offline cache, and pending progress.
3. Open the app once and verify the version (§5.5).
4. Update the operator's device/version log so the next backend CONTRACT
   CHANGE conversation knows who runs what.

### 5.3 Android blocks unknown-app installs

Modern Android gates sideloads per *delivering* app, not globally:

- The install prompt usually offers **Settings** directly; toggle
  **Allow from this source** for the app the APK was opened from (Files,
  Chrome, or the messaging app), then go back and Install.
- Manual path: Settings → Apps → Special app access → **Install unknown
  apps** → choose the delivering app → allow.
- The toggle can be turned back off after installing; it is per-source and
  per-device, one-time friction on first install only.
- If a work-managed device policy blocks sideloads entirely, that device
  cannot run the household app for now — same stance as the Tailscale
  restriction in GETTING_STARTED; do not fight device management.

### 5.4 Install fails: signing mismatch or lower versionCode

- **"App not installed"** on an update usually means one of:
  - **Signature mismatch** — the device has a build signed with a different
    key (typically an old debug/Android-Studio install from the pre-signed
    era, or a lost-key rebuild). Fix: uninstall the existing app, then install
    the new APK. Warn the member first: device-local cache/pending items are
    lost; library and progress return after sign-in. Expected exactly once per
    device when migrating from a dev install to the signed line.
  - **Downgrade attempt** — the APK's `versionCode` is lower than what's
    installed. Android never downgrades in place. Fix: distribute a build with
    a higher code; if a genuine rollback is needed, bump `versionCode` on the
    old source and rebuild (the roll-forward rule), or uninstall/reinstall
    with the data-loss caveat.
- Operator diagnosis, no Android Studio needed: check the installed version
  via §5.5; check the APK's own version/signature on the build machine with
  the SDK build-tools (`aapt dump badging <apk>` /
  `apksigner verify --print-certs <apk>`) and compare the certificate
  fingerprint to the one recorded in `RELEASE_NOTES.md`.

### 5.5 Verifying the installed version

- **Member-friendly path (today):** Android Settings → Apps → Mimeo → the
  version string (`versionName`) is shown in App info.
- **In-app (after T-AND-DIST-MIN-1):** the app's Settings screen shows an
  "App version" line — `versionName (versionCode)` — so "read me what
  Settings says" works over the phone. (Today the app surfaces version only
  inside problem-report payloads, which is not member-checkable.)
- **Operator path with a cable:** `adb shell dumpsys package com.mimeo.android
  | findstr version` shows both `versionCode` and `versionName`.

## 6. Pilot readiness — what T-AND-DIST-MIN-1 must prove before T-HH-PILOT-1

The pilot script's preflight (§2) and stop-condition 4 hard-require this
ticket. T-AND-DIST-MIN-1's acceptance evidence, all logged in the PR:

1. **Signed APK built** — `assembleRelease` output verified signed with the
   household key (`apksigner verify --print-certs`), fingerprint matches the
   one recorded in `RELEASE_NOTES.md`.
2. **Installed on a non-developer Android device** — a real device that is not
   the operator's development target, via the §5.1 runbook exactly as written
   (friction found = runbook edit in the same PR).
3. **Update over previous signed build tested** — install signed build N,
   then N+1 (higher `versionCode`) over the top; sign-in and local state
   survive. This is the review's exit criterion for queue item 4 verbatim.
4. **App connects to the Tailscale endpoint** — with the device on the
   tailnet, connection test against
   `https://beh-august2015.taildacac5.ts.net` succeeds in the release build
   (this also re-proves the release network-security config allows the HTTPS
   tailnet host — release builds deny cleartext to public hosts by design).
5. **Sign-in succeeds** — username/password sign-in on the release build (a
   test/`pilot-` account is fine; no real member required yet).
6. **No Android Studio needed during member install** — the install/update
   steps above executed from the APK artifact alone; the runbook contains no
   step that touches a dev machine on the member side.

Anything less and T-HH-PILOT-1 must not start (pilot stop condition 4).

## 7. Risks

| # | Risk | Stance / mitigation |
|---|---|---|
| 1 | **Sideload friction** — unknown-sources prompts, Play Protect warnings scare members or stall installs | Operator performs/assists every install (household-ready review I7 posture); §5.3 documents each prompt so it is expected, not alarming; friction from the proof-install feeds runbook edits |
| 2 | **Key loss** — keystore or passwords gone → no in-place updates ever again | Two-copy backup rule + escrow rhythm (§3); recovery path documented (uninstall/reinstall, server data survives); fingerprint recorded so key identity is verifiable |
| 3 | **Signing mismatch** — leftover debug installs on early devices reject signed updates | Expected one-time migration per device; §5.4 documents uninstall-once fix and the data-loss warning; debug APKs never distributed to members from now on |
| 4 | **versionCode mistakes** — forgotten bump ships an un-installable "update"; reused code masks which build a device runs | §4 discipline: bump in every release PR, release notes record code, operator keeps per-device version log; T-AND-DIST-MIN-1's update test catches the failure mode once |
| 5 | **Members ignoring update instructions** — household devices drift across versions | Accepted at this tier: updates are operator-delivered in person, not member-initiated; per-device version log shows drift; backend contract notes (§4) say when drift actually matters |
| 6 | **Tailscale still required** — app installs fine but connects nowhere without tailnet membership; members may blame the app | Out of this ticket's scope to fix (public ingress is its own gated decision); runbook step 5.1-1 orders Tailscale first; GETTING_STARTED already frames it as a hard requirement |
| 7 | **APK leakage** — artifact forwarded outside the household | Low impact (app is useless off-tailnet without credentials) but the rule stands: trusted channels only, never public links (§5) |
